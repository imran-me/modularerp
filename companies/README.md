# companies/ — every sister concern is a self-contained folder

> Delete a company folder → that company disappears (fully wired in Phase 3).
> Drop it back → it returns. See `EPAL_GROUP_ERP_Modular_Architecture.md`.

## Anatomy of a company

```
companies/<company>/
├── module.json            ← the company MANIFEST: identity, accent, and the FULL
│                            menu tree (modules + sub-menus + routes). Phase 3
│                            auto-discovery reads this to build the sidebar.
├── app/
│   ├── atmosphere/        ← the company's OWN ambient animation scene (css+js)
│   └── frontend/          ← standalone HTML+Tailwind pages (Travels has its
│                            multi-page mini-app here)
└── modules/<module-id>/   ← one folder per BUILT module
    ├── module.json        ← the module manifest: title, route, sub-menus, roles
    ├── view.js            ← the module's ENTIRE frontend. In this SPA the HTML
    │                        is generated here (ui.el('div.card' …) calls) — this
    │                        file IS the html+js of the screen. Its styling
    │                        converts to Tailwind utility classes in Phase 4,
    │                        written into these same el() calls.
    └── backend/
        └── LARAVEL-BLUEPRINT.md ← the module's backend, as the spec a Laravel
                                   developer builds 1:1: entities & fields,
                                   business rules, routes, controllers, models &
                                   migrations, policies, domain events (for the
                                   group bridge), engine dependencies.
```

## Where each technology lives (and why)

| You asked for | Where it is | Why |
|---|---|---|
| **HTML** | Generated by each module's `view.js` (SPA), plus real `.html` pages in `app/frontend/` | The app is a single-page app — screens are built in JS, exactly as before |
| **CSS / Tailwind** | Shared design system: `platform/design-system/css/`; Tailwind lock: `platform/design-system/tailwind.config.js`. Per-module `tw-` utility classes arrive inside each `view.js` during **Phase 4** | One design system = every company looks consistent; per-company accents are injected at runtime |
| **JS** | `modules/<id>/view.js` per screen; shared machinery in `platform/` | Deliberately readable ES5-style IIFEs — the code doubles as the Laravel handoff spec (owner directive); converting to ES6 modules would require a build step and risk behaviour |
| **Backend** | `modules/<id>/backend/LARAVEL-BLUEPRINT.md` per module — plus the working demo "backend" (localStorage data + engines) in `platform/data/` + `platform/engines-library/` | GitHub Pages cannot run PHP/servers (free static hosting, owner directive). The REAL backend is what the Laravel developer builds FROM the blueprint; the shared engines are the architecture doc §6 "shared machinery" option |
| **Data** | `platform/data/` (one shared store) — per-company `data/` split is deliberately deferred | The Phase-0 inventory proved group totals depend on the shared store's exact shape; splitting it before the Phase-3 bridge would change the numbers |
| **Animations** | `app/atmosphere/` inside the owning company | Kept as scoped CSS per the owner's Phase-4 ruling ("if it ruins the animation, keep that format") |

## Scaffold-only modules

Modules that exist in the menus but don't have a bespoke screen yet (they render
via the shared wildcard views or the designed placeholder) do NOT have a folder
here — they are declared in the company-level `module.json`. When one gets
built, it materialises as `modules/<id>/` with the same anatomy.
